# 编译部署手册

## 1. 目录说明

```text
license_product_source_final_v2/
├── license-server/       # 授权服务端
├── electron-client/      # 宜搭 Electron 客户端
├── scripts/              # 辅助脚本
├── docker-compose.yml    # 服务端容器部署
└── README.md
```

## 2. 服务端部署

### 2.1 准备环境

服务器建议：

```text
Debian / Ubuntu
Docker
Docker Compose Plugin
```

安装 Docker 后确认：

```bash
docker version
docker compose version
```

### 2.2 生成生产密钥

源码包里自带一套开发密钥，只是为了让代码开箱能跑。

正式上线前必须重新生成：

```bash
cd license_product_source_final_v2
./scripts/generate_rs256_keys.sh
```

生成后：

```text
license-server/keys/private.pem   # 服务端私钥，严禁泄露
license-server/keys/public.pem    # 服务端验签公钥
electron-client/keys/public.pem   # 客户端内置公钥
```

### 2.3 配置环境变量

```bash
cp license-server/.env.example license-server/.env
nano license-server/.env
```

推荐配置：

```env
APP_NAME=License Server
DATABASE_URL=sqlite:///./data/license.db
JWT_ALGORITHM=RS256
PRIVATE_KEY_PATH=/app/keys/private.pem
PUBLIC_KEY_PATH=/app/keys/public.pem
ADMIN_USERNAME=admin
ADMIN_PASSWORD=请改成你的后台密码
ALLOW_INSECURE_DEFAULTS=false
```

> V2 已改用 pbkdf2_sha256，不再受 bcrypt 72 字节密码限制。
> 服务端会拒绝使用 `admin123456`、`change-me` 等默认弱密码启动；只允许在短期本地开发时显式设置 `ALLOW_INSECURE_DEFAULTS=true`。

### 2.4 启动服务端

```bash
sudo docker compose up -d --build
sudo docker logs -f license-server
```

看到下面日志表示成功：

```text
Application startup complete
```

### 2.5 访问后台

```text
http://服务器IP:8000/admin
```

登录后可执行：

```text
创建卡密
查看卡密
启用 / 禁用
续期
查看设备
解绑设备
查看激活日志
```

## 3. 反向代理和 HTTPS

建议用 Nginx 做 HTTPS。

示例：

```nginx
server {
    listen 443 ssl;
    server_name license.example.com;

    ssl_certificate /etc/letsencrypt/live/license.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/license.example.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

客户端接口地址应写成：

```text
https://license.example.com/api/v1
```

## 4. 客户端开发运行

### 4.1 安装依赖

```bash
cd electron-client
npm install
```

### 4.2 配置授权服务地址

开发时可用环境变量。

Windows PowerShell：

```powershell
$env:AUTH_SERVER="https://license.example.com/api/v1"
npm start
```

Windows CMD：

```cmd
set AUTH_SERVER=https://license.example.com/api/v1
npm start
```

macOS / Linux：

```bash
AUTH_SERVER=https://license.example.com/api/v1 npm start
```

如果不设置，默认是：

```text
http://127.0.0.1:8000/api/v1
```

### 4.3 配置正式打包地址和本地授权密钥

正式打包前必须编辑 `electron-client/package.json`：

```json
"config": {
  "authServer": "https://license.example.com/api/v1",
  "storageSecret": "替换为随机长字符串"
}
```

- `authServer` 是客户端激活和在线校验的授权服务地址。
- `storageSecret` 用于加密本地授权文件，建议使用随机生成的长字符串。
- 开发运行时仍可用 `AUTH_SERVER` 或 `CLIENT_STORAGE_SECRET` 环境变量临时覆盖。

### 4.4 宜搭登录结果

客户端激活后会打开宜搭登录页。

登录成功后会自动保存：

```text
桌面/yida-login-result/.cache/cookies-public.json
```

文件包含：

```json
{
  "cookies": [],
  "base_url": "https://www.aliwork.com"
}
```

## 5. 客户端打包

### 5.1 Windows 打包

```bash
cd electron-client
npm install
npm run build:win
```

输出目录：

```text
electron-client/dist/
```

默认输出：

```text
yida-self-login-版本号-x64.exe
```

### 5.2 macOS 打包

```bash
npm run build:mac
```

默认输出：

```text
yida-self-login-版本号-x64.dmg / .zip
yida-self-login-版本号-arm64.dmg / .zip
```

### 5.3 Linux 打包

```bash
npm run build:linux
```

默认输出：

```text
yida-self-login-版本号-x64.AppImage
yida-self-login-版本号-x64.deb
```

### 5.4 跨平台建议

- 最稳妥方式是在对应系统上打对应平台包：Windows 打 Windows，macOS 打 macOS，Linux 打 Linux。
- 仓库已提供 GitHub Actions 工作流：`.github/workflows/build-clients.yml`。在仓库 Secrets 中配置 `AUTH_SERVER` 和 `CLIENT_STORAGE_SECRET` 后，可手动触发或推送 `v*` tag 自动构建三平台产物。
- macOS 正式分发需要 Developer ID 证书和 notarization；否则只能 ad-hoc 签名，用户首次打开可能看到安全提示。
- Windows 正式分发建议配置代码签名证书，减少 SmartScreen 拦截。
- Linux AppImage 可直接运行，deb 适合 Debian/Ubuntu 系发行版。
- 所有 `build:*` 命令都会先检查 `authServer` 和 `storageSecret`，缺配置会直接失败，避免产出不可激活的客户端。

## 6. 生产发布流程

推荐顺序：

```text
1. 生成生产 RS256 密钥
2. 复制 license-server/.env.example 为 .env
3. 修改 ADMIN_PASSWORD
4. 部署服务端
5. 登录后台创建测试卡密
6. 确认 electron-client/keys/public.pem 是最新公钥
7. 修改 electron-client/package.json 中的 config.authServer 和 config.storageSecret
8. 打包客户端
9. 使用测试卡密激活
10. 确认宜搭 Cookie 文件可正常生成
```

## 7. 常见问题

### 7.1 服务端仍然报 bcrypt 错误

V2 已经不依赖 bcrypt。如果还有 bcrypt 报错，说明你跑的不是 V2 代码，或者 Docker 没有重新 build。

执行：

```bash
sudo docker compose down
sudo docker compose build --no-cache
sudo docker compose up -d
```

### 7.2 客户端提示“授权文件无效”

通常是服务端私钥和客户端公钥不匹配。

解决：

```bash
./scripts/generate_rs256_keys.sh
sudo docker compose up -d --build
cd electron-client
npm run build:win
```

并清理旧授权文件：

```text
Windows: C:\Users\你的用户名\AppData\Roaming\yida-electron-self-login\license.dat
```

### 7.3 客户端请求 404

确认地址必须带 `/api/v1`：

```text
https://你的域名/api/v1
```

激活接口实际是：

```text
POST /api/v1/license/activate
```

### 7.4 后台密码改了但登录不上

如果数据库已经初始化过，`.env` 修改不会自动覆盖已有管理员密码。

测试环境可删除数据库：

```bash
sudo docker compose down
rm -f license-server/data/license.db
sudo docker compose up -d --build
```

生产环境不要直接删库，应新增“重置管理员密码”脚本后再操作。

## 8. 安全注意事项

- `license-server/keys/private.pem` 不能发给客户。
- 客户端只能包含 `public.pem`。
- 打包前必须修改 `electron-client/package.json` 中的 `config.storageSecret`。
- `license-server/.env`、`license-server/data/`、`license-server/keys/private.pem` 不应提交到 Git。
- Electron 无法做到绝对防破解，只能提高破解成本。
- 真正高价值逻辑应尽量放在服务端。
